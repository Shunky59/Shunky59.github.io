MEDS: Autonomous Delivery Robot

Senior Capstone Design: A fully autonomous system for hospital medication delivery.

Systems Engineering Python & Flask Mechatronics

The Challenge

For our Senior Design Capstone, our team designed, built, and validated the Medical Express Delivery System (MEDS). The objective was to create an autonomous robot (1 ft √ó 1 ft) capable of navigating a hospital environment, detecting "patient" distress signals via wearable sensors, and dispensing medication.

The complete system: Autonomous Robot, Koozie Docking Station, and Patient Wearables.

Our final system featured a skid-steering drivetrain for zero-radius turns, a custom rack-and-pinion dispenser, and a sophisticated fever-detection algorithm. We completed the build for $836.14, staying well under our $1,000 budget.

Electrical & Systems Architecture

As the electrical and software lead, I designed a centralized control architecture powered by a Raspberry Pi 4B. Unlike simpler microcontroller-based robots, the Pi allowed us to run a full Linux operating system to handle complex networking and multi-threading tasks.

The Pi acts as a WiFi Hotspot, creating a dedicated local network. The patient wearable devices (powered by ESP32 microcontrollers) connect directly to this network to transmit real-time biometric data (temperature and button presses) via JSON packets. This dedicated network ensured low-latency communication independent of external hospital WiFi infrastructure.

On the robot side, the Pi interfaces with:

  • RoboClaw Motor Controller: Handles high-current power distribution and PID velocity control for the drivetrain.
  • BNO055 IMU: Provides absolute heading (yaw) data via I2C, critical for the robot's dead-reckoning navigation algorithms.
  • Ultrasonic Sensors: Used for obstacle avoidance and the precision "sweeping scan" docking algorithm.
  • OLED Display: Outputs real-time system status, current state (e.g., "Scanning", "Delivering"), and IP addresses for debugging.

Software Architecture & Algorithms

I developed the robot's control stack entirely in Python. The complexity of handling asynchronous network traffic while simultaneously driving the motors required a robust multi-threaded architecture.

1. Threading & Network Logic

The system runs two primary threads. Thread 1 hosts a Flask Web Server that listens for incoming HTTP POST requests from the wearables. Thread 2 runs the main navigation loop. This separation ensures that a delay in network traffic never causes the robot's control loop to hang or stutter.

2. Wearable Signal Processing

The wearable code (C++) implements a median filtering algorithm to clean the raw data from the thermistor. It takes a rapid burst of temperature readings, sorts them, and discards outliers to prevent false "fever" triggers from noise.

3. Navigation Strategies: "Scanning" vs "Fast"

I developed two distinct navigation strategies:

  • Scanning Mode: Uses a state-machine approach. If the robot arrives at the patient room but doesn't immediately align with the Koozie, it enters a "Sweep" state, using the ultrasonic sensor to scan the area for the specific geometric profile of the docking station before dispensing.
  • Fast Mode (Dead Reckoning): For speed optimization, this mode relies on a highly tuned sensor fusion model (IMU + Encoders) to drive precise paths without stopping to scan, relying on the mechanical accuracy of our drivetrain.

Mechanical Integration & Prototyping

The mechanical design supported the electrical needs. We utilized a skid-steering drivetrain for zero-radius turning, allowing agility in tight hospital corridors. We validated this design early using a MATLAB trade study to ensure our motors provided sufficient torque.
